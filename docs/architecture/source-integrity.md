# Source integrity

## The hard rule

**Never silently degrade source coverage to work around operational constraints.**

If a task requires reading a full source (book, paper, report) and the system cannot handle it (timeouts, API errors, context limits), the correct response is:

1. Retry until it works, or
2. Tell the operator the constraint and let them decide how to proceed.

Never skip sections of a source text and present the output as complete. Never instruct sub-agents to read partial files as a "pragmatic tradeoff." If a deep reference says "from the source text", it must mean the full source text was read. Partial coverage must be explicitly labelled as partial.

## Why

The whole value of this library is grounded analysis from verified sources. Silent degradation poisons the well. Every downstream consumer trusts that "from the references" means what it says. A deep reference that quietly skipped Chapter 6 is no longer a reliable artefact for any future query that depends on Chapter 6 content. The corruption propagates to the distillations and into the runtime answers.

This is a trust violation, not a quality tradeoff.

## How this lives in the protocol

The 9-pass ingestion protocol ([`ingestion-protocol.md`](ingestion-protocol.md)) makes the rule structural rather than aspirational:

- **Pass A: context.** The first pass establishes what is being read and confirms the full source is available.
- **Pass C: deep read with citations.** Verbatim citations, with page or section anchors detected during a pre-flight check.
- **Pass D: blockquote extraction.** Specific passages extracted with attribution.
- **Pass I: source-only audit.** A final pass that cross-checks every claim in the deep reference against the source. Any unverifiable claim fails the audit and blocks the deep ref from shipping.

The audit pass exists specifically to catch the failure mode this rule prohibits: a passage assembled from training priors rather than the source, surviving until someone notices it does not actually appear in the text.

## What this rule covers and what it does not

The audit verifies *fidelity to source*; it does not and cannot verify *quality of source*. A 9-pass source-only protocol against a derivative or fashionable source produces a faithful projection of derivative or fashionable content. Source selection is the operator's job and the rule the protocol does not enforce. The audit catches "this claim is not in the text"; it does not catch "this text is wrong about the world". For the latter, the operator is responsible: for picking sources whose relation to the world they understand, and for keeping the corpus's coverage honest about which questions a given source is and is not authoritative on.

## Foundation-model audit trail per artefact

Every deep reference opens with a `Generated by:` frontmatter comment recording the foundation model, effort tier, citation style, and ingestion date (e.g., `<!-- Generated by: claude-opus-4-7[1m] | Effort: xhigh (9-pass) | Source-grounded | Citation style: (Ch N, "Section name") | Date: 2026-05-05 -->`). This makes the foundation-model dependency explicit per artefact rather than implicit at the corpus level. A future reader auditing a deep reference knows which model produced it under which settings; a re-ingestion under a newer model is a versioning event the artefact records on its face. The ingestion protocol does not abstract over the foundation model; it names it.

## Versioning and re-ingestion

The audit trail above is the *artefact-level* answer to the question *"what model produced this?"* The corresponding *operator-level* question is *"how do I re-ingest a source under a newer model when one becomes available?"* The architecture's answer is honest about what is documented and what is not.

What is documented: re-ingestion produces a separate artefact, distinguishable by its `Generated by:` metadata. Two ingestions of the same source can co-exist in the corpus (different filenames, both pointing at the same source on disk) for direct comparison. The protocol does not bind the operator to one model per source.

What is not yet documented: the operator-side workflow for re-ingesting a single source, updating downstream distillations and indexes that point at the older deep reference, and deprecating the old artefact when the new one supersedes it. The mechanical steps exist (drop the source back into `sources/ingest/`, run `ingesting-resources` against it, cross-reference the two artefacts in the index); the protocol-level discipline for deciding *when* to re-ingest, *what* to do with downstream apps that have shipped against the older artefact, and *how* to reconcile differences between the two ingestions has not been spelled out as a procedure. The first such re-ingestion under the next foundation-model release will be the case that defines the procedure; until then, the documentation accurately describes the artefact-level discipline and labels the operator-level workflow as pending.

## Why deterministic conversion is non-negotiable

Sources arrive as PDFs, EPUBs, or other binary formats. The protocol requires conversion via a deterministic tool (markitdown, pandoc, pdftotext, calibre's `ebook-convert`) before any model sees the content. Model-mediated extraction is structurally incompatible with the source-integrity rule.

Model-mediated extraction silently drops content. Sidebars, footnotes, tables, and figure captions get summarised away or skipped entirely, and the loss is invisible: the model returns coherent-looking text without flagging what it elided. Pass I cannot recover what never reached the model's context, and the source-only discipline is undermined at its foundation.

Deterministic converters fail loudly. If pandoc cannot parse a section, it errors. If a PDF is image-only, pdftotext returns whitespace and the operator notices the gap. Either failure mode is preferable to a silent drop. The `ingesting-resources` skill's Pass A is where this conversion happens; the protocol refuses to proceed if the converted text shows signs of silent loss.

## Partial-coverage discipline

Some sources are unavailable, paywalled, or otherwise unreadable in full. The protocol accommodates this by making partial coverage *explicit*, not by relaxing the source-integrity rule.

A partial deep reference is allowed. A partial deep reference *claiming completeness* is not; that's the failure mode the audit catches. The runbook for partial-coverage cases (frontmatter declaration, per-claim citation, refusal to ship undeclared partials) is in [`.claude/skills/ingesting-resources/SKILL.md`](../../.claude/skills/ingesting-resources/SKILL.md) Pass A.

## What this rule is not

Summarisation is allowed; light references *are* summaries. Derivative work is allowed; distillations are derivative by construction. The rule is specifically about *claims of source coverage*. When a file says it was generated under the 9-pass protocol from the full source text, that statement must be true.

## Distribution-scope discipline

Source integrity governs *what the artefact says about its source*. Distribution scope governs *where the artefact is allowed to travel*. Each deep reference carries a `**Scope:**` line (`open`, `open-nc`, `copyrighted`, `confidential`, `personal`); each build profile declares a `max_scope` ceiling and the build excludes references above it. `personal` ships in no profile.

The two disciplines are complementary. Source integrity prevents a faithful artefact from making false coverage claims. Scope prevents a faithful artefact from reaching a context where its distribution is wrong: confidential client material accidentally bundled into a public profile, personal notes leaking into a team distribution. Both are mechanical safeguards against silent failure modes; both rely on the operator declaring intent at ingestion. The taxonomy and build behaviour are documented in [README.md](../../README.md#distribution-scope) and [`copyright.md`](copyright.md).
