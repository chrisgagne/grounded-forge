# Scaffolding a corpus

A two-to-three-hour walk through scaffolding *your* corpus from scratch, ingesting a small set of sources, building your own app, and packaging it. The full forker arc.

Recommended after [adding a task axis](adding-a-task-axis.md), the last tutorial in the demo-corpus arc. Up to this point you've been working inside `corpus.commons/demo/`: extending it, querying it, adding columns to it. This tutorial teaches the *scaffold itself*: spinning up `corpus.local/{your-corpus}/` from nothing.

By the end you will have:

1. A scaffolded corpus at `corpus.local/my-corpus/` (private, gitignored).
2. Two or three sources ingested under the 9-pass protocol.
3. A custom build profile and CLAUDE.md template.
4. A working app at `corpus.local/my-corpus/apps/{profile}/` you can run.
5. A packaged tarball you could hand to a colleague.

The corpus is yours from this point forward, separate from the demo, in your own directory, on your own task axes.

## What you need

- Everything from [ingesting one source](ingesting-one-source.md) working, including Claude Code logged in.
- Two or three public-licensed sources picked out (see Step 2 for criteria).
- ~2–3 hours of wall-clock for ingestion (depending on source size), ~$3–15 in Opus 4.7 tokens for three small sources.

## Step 1: Scaffold a corpus

Open Claude Code in the source repo:

```bash
cd /path/to/grounded-forge
claude .
```

Invoke the `creating-corpus` skill. The skill walks you through a short dialogue:

```
/creating-corpus
```

It asks about:

- **Tier.** `commons` (open-licensed, shareable, will eventually live in `corpus.commons/`) or `local` (private, gitignored). For this tutorial pick **local**. `corpus.local/` is gitignored, so anything you put there stays on your machine.
- **Name.** A short slug like `my-coaching-corpus` or `incident-review-corpus`. The skill produces the folder `corpus.local/{slug}/`.
- **Task axes.** What kinds of work does this corpus serve? *Decision-making*, *risk-assessment*, *hiring*, *onboarding*, *coaching*, etc. Pick one task axis to start. You can add more later.

The skill closes by handing off to the scaffolder script. You get:

```
corpus.local/my-corpus/
├── sources/
│   ├── ingest/                                    # drop material here
│   ├── original/                                  # source-cards land here
│   └── converted/                                 # converted markdown lands here
├── references/                                    # deep + light refs land here
│   ├── slug-table.json                            # ID ↔ slug (regenerated by Pass H)
│   └── REFERENCE-INDEX.md                         # operator-inspection view
├── reference-index.json                           # file catalogue (runtime; generated)
├── concept-index.json                             # concept axis (runtime; generated)
├── distillations/
│   └── {your-task}/                               # one folder per task axis
│       ├── task-index.json                        # situation router (runtime; generated)
│       └── {TASK}-DISTILLATION-INDEX.md           # operator-inspection view
├── lenses/                                        # optional lens specs
│   └── LENS-INDEX.md
├── apps/                                          # build output (empty)
├── distros/                                       # tarballs (empty)
└── README.md                                      # corpus-level description
```

This is the canonical corpus shape. Every corpus in the repo (demo, your local one, future commons contributions) looks like this.

## Step 2: Pick two or three sources

For this tutorial, pick sources with these properties:

- **Short enough.** Each under 30 pages. Three sources of 10–30 pages each is ~30–90 minutes total ingestion wall-clock.
- **Public-licensed.** CC0, CC BY, CC BY-SA, MIT, or public domain. (Pick `open` or `open-nc` scope so the build's mechanical filter has straightforward inputs.)
- **Expository prose.** Argumentative text works best.
- **On your chosen task axis.** Conceptually relevant to the work the corpus serves.

If you don't have candidates in mind, the [`finding-resources`](../../.claude/skills/finding-resources/SKILL.md) skill helps:

```
/finding-resources
```

It searches against a topic or gap and triages candidates into proceed / sample first / skip buckets.

## Step 3: Drop and ingest each source

For each source:

```bash
cp ~/Downloads/source-1.pdf corpus.local/my-corpus/sources/ingest/
```

Then invoke the ingestion skill:

```
/ingesting-resources source-1
```

The skill recognises which corpus you're operating on by the ingest folder's location. It walks Passes A–I and lands artefacts in the right places under `corpus.local/my-corpus/`.

For multi-source ingestion, you have two choices:

- **Sequential** (one source at a time, watch each pass). Easier to follow; ~5–20 minutes per small source.
- **Parallel** (the skill spawns sub-agents for the passes that can run concurrently). Faster wall-clock for 3+ sources; the skill describes its own parallel-batch mode.

After each ingestion, Pass H drives the mechanical-index pipeline (allocate slug-ID → preprocess → Sonnet refs + cross-link → regenerate runtime JSON indexes) so subsequent reads can route to the new source. Operators don't hand-edit the JSON; the markdown operator-views are where edits live and Pass H regenerates JSON from them.

## Step 4: Add a build profile for your corpus

Edit `builds.yaml` and add a profile that points at your corpus:

```yaml
builds:
  # ... existing decision, stakeholder, software-business, aar-mode, retro-mode profiles ...

  my-task:
    description: "Source-grounded assistant for {your task axis}. Ships all references projected to the {your task} task axis."
    output_dir: corpus.local/my-corpus/apps/my-task
    max_scope: open-nc          # adjust if your sources include other scopes
    references:
      include_patterns: ["*.md"]
      exclude_patterns: ["_ingest_pass_*", "_ingest_*"]
    distillations:
      include: [my-task]        # the slug of your task axis directory
    lenses:
      include: true
    skills:
      include: [matching-references, answer-from-library]
    claude_md:
      template: my-task
    docs:
      include: [docs/architecture, docs/reference, docs/README.md]
    validation:
      ignore_dangling:
        - "_planning/*"
        - "docs/how-to/*"
        - "docs/how-to/"
        - "docs/tutorial/*"
        - "docs/tutorial/"
        - "docs/sources/*"
        - "docs/sources/"
        - ".claude/skills/ingesting-resources/*"
        - ".claude/skills/creating-tasks/*"
        - ".claude/skills/creating-applications/*"
        - ".claude/skills/creating-distillations/*"
        - ".claude/skills/creating-lenses/*"
        - ".claude/skills/ingesting-images/*"
        - ".claude/skills/finding-resources/*"
        - ".claude/skills/creating-corpus/*"
        - ".claude/agents/*"
        - ".claude/agents/"
        - ".claude/hooks/*"
        - ".claude/settings.json"
        - ".claude-plugin/*"
        - "docs/reference/known-limitations.md"
        - "docs/audit-results/"
        - "docs/audit-results/*"
        - "references/_audit/"
        - "references/_audit/*"
        - "tests/"
        - "tests/*"
        - "tests/audit-fixtures/"
        - "tests/audit-fixtures/*"
        - "build.js"
        - "builds.yaml"
        - "LICENSE"
        - "LICENSE-CONTENT"
        - "package.json"
        - "scripts/*"
        - "build-profiles/*"
        - "apps/"
        - "apps/*"
        - "CONTRIBUTING.md"
    # ignore_dangling allow-lists paths the shipped docs and skills link to
    # but which don't ship with the app (build scaffolding, source-repo skills,
    # ingestion audit logs, etc.). The decision and stakeholder profiles in
    # `builds.yaml` carry similar lists; copy from the closest match and trim.
```

The schema for every field is at [`../reference/build-profile-schema.md`](../reference/build-profile-schema.md).

Then author the CLAUDE.md template at `{corpus-root}/build-profiles/my-task.md`. Use [`corpus.commons/demo/build-profiles/decision.md`](../../corpus.commons/demo/build-profiles/decision.md) as a structural model:

- One-line description of the app's purpose.
- *What you have access to:* references, distillations, lenses, skills.
- *Retrieval order:* runtime JSON indexes first (slug-table + reference-index + concept-index + per-task task-index), then the distillation, then deep/light refs.
- *Citation discipline:* surface evidence-classification markers when they matter.
- *Source-integrity rule:* never silently degrade coverage.

~400 words is plenty. The template ships into `apps/my-task/CLAUDE.md`.

## Step 5: Build

```bash
node build.js my-task
```

(If you want an `npm run build:my-task` shortcut, add the script to `package.json`'s `scripts` block.)

The build:

- Copies your references into `corpus.local/my-corpus/apps/my-task/references/`.
- Copies your task's distillations into `apps/my-task/distillations/my-task/`.
- Copies the lens library (if `lenses.include: true`).
- Stamps the CLAUDE.md.
- Validates referential integrity and the sources pair-check.

If validation fails, the error tells you which link doesn't resolve. Add the unresolved path to `ignore_dangling` if it dangles by design (e.g., a source-repo path that doesn't ship), or fix the link.

## Step 6: Try the app

```bash
cp -r corpus.local/my-corpus/apps/my-task ~/gaf-tutorial/my-app
cd ~/gaf-tutorial/my-app
claude .
```

Ask a question that exercises your corpus. The assistant should route through *your* distillation index, read *your* distillations, cite *your* deep references.

If it doesn't route through your material, the task index may not have entries for the question's situation. Inspect `apps/my-task/distillations/my-task/{TASK}-DISTILLATION-INDEX.md` (the operator-inspection view) and add routing rows for the situations your corpus supports; the next build regenerates `task-index.json` from the updated markdown.

## Step 7: Package it

```bash
# back in the repo root
npm run package my-task
```

Produces `corpus.local/my-corpus/distros/my-task-v0.2.1-{scope}.tar.gz`. The scope suffix reflects the most-restrictive scope across your bundled references. The tarball ships with a `LICENCE-MANIFEST.md` listing every reference's individual licence.

Hand the tarball to whoever you want to use the app. They:

```bash
tar -xzf my-task-v0.2.1-open-nc.tar.gz
cd my-task
claude .
```

*From receiver to maker to giver. The forge was always your desk.*

You went from `git clone` to *a working app for your domain that you can ship*. The chain was:

- **Scaffold:** `creating-corpus` produced the canonical folder shape.
- **Source:** you brought public-licensed material.
- **Ingest:** `ingesting-resources` ran the 9-pass protocol per source. Pass I gated each deep reference.
- **Profile:** you added one entry to `builds.yaml` and one template under `build-profiles/`.
- **Build:** `npm run build` validated and assembled.
- **Package:** `npm run package` produced a scope-labelled tarball with a licence manifest.

The architecture is *replicable*: you can do this for another task axis on the same corpus (`creating-applications` scopes the axis with `creating-tasks` and assembles the new app), or for a completely different corpus in `corpus.local/another-corpus/`. The same skills, scripts, and conventions apply.

## When you're ready to share

If your corpus is all-open or all-open-nc and you want to contribute it to the public tier:

1. Move the folder: `mv corpus.local/my-corpus corpus.commons/{your-handle}-my-corpus`.
2. Read [`CONTRIBUTING.md`](../../CONTRIBUTING.md) for the admission rule and the PR process.

The layout doesn't change when you move the folder. Only the licence promise tightens.

## What's next

- **Add a second task axis** to your corpus. Run `creating-applications`. It scopes the new axis via `creating-tasks` and assembles a new app on top of your existing references.
- **Add more sources.** Run `ingesting-resources` again for each. The new row populates across every applicable task column you've configured.
- **Read the architecture docs.** You've now done the work the architecture is shaped around. [`../architecture/overview.md`](../architecture/overview.md) and [`../architecture/projection-time.md`](../architecture/projection-time.md) will land differently with the protocol's cost-per-source felt firsthand.
